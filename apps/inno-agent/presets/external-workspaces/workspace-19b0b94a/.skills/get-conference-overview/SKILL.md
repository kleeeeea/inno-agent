# 获取会议概览
## When to use
- 用户询问会议的整体主题、背景或主要内容时。
- 用户想知道会议的具体日期和时间时。
- 用户查询会议的举办地点时。
- 用户需要了解会议的整体议程或流程安排时。

## When not to use
- 用户询问特定时间段的活动内容、主讲嘉宾或圆桌讨论主题时，应路由至 `query-session-details`。
- 用户询问会议室的具体名称（因文档中注明“待定”），应告知信息缺失。
- 用户请求会议报名、联系嘉宾、预定会议室、交通指引等实时互动或操作时，应拒绝并说明超出能力范围。
- 用户询问会议是否有线上直播、提供会议资料或录像等文档中未提及的信息时，应告知信息不在知识范围内。
- 用户请求任何嘉宾的联系方式或其他个人隐私信息时，应拒绝。

## Instructions
- 严格根据提供的《中外高校数智化教育合作交流会议程》文档，提取会议的整体主题、时间、地点及主要流程信息。
- 对于文档中明确注明“待定”或未提及的信息，如实告知用户信息缺失，不进行任何猜测或补充。
- 确保输出信息的准确性、简洁性和完整性。

## Workflow
1.  **理解用户意图**: 判断用户是否寻求会议的整体概览信息（主题、时间、地点、主要流程）。
2.  **信息检索**: 从会议议程文档中提取“主题”、“时间”、“地点”以及各主要环节的概述。
3.  **结果呈现**: 以清晰、简洁的中文文本形式向用户提供会议概览信息。
4.  **处理信息缺失**: 如果用户询问的信息在文档中明确缺失（如具体会议室），则告知用户该信息尚未确定。
5.  **引导与澄清**: 如果用户请求不明确或超出本技能范围，进行澄清或引导至其他技能。

## Output Format
- **场景判断**: 用户请求会议整体概览信息。
- **处理方案**: 从文档中提取会议主题、时间、地点及主要流程，并以结构化方式呈现。
- **可用话术**:
    - “本次会议的主题是：[会议主题]。”
    - “会议时间定于：[会议时间]。”
    - “会议地点在：[会议地点]，具体会议室待定。”
    - “会议的整体安排包括：[环节一]、[环节二]、[环节三]等。”
- **风险提示**:
    - 避免对“具体会议室待定”等信息进行猜测。
    - 避免提供超出文档范围的信息。
- **后续评估**: 用户是否获得了准确、完整的会议概览信息。

## Examples

**Example 1: 用户询问会议主题**
- **User Request**: “这次会议主要讲什么？”
- **Agent Response**: “本次‘中外高校数智化教育合作交流会议’的主题是：数智赋能 · 融创未来 ——AI与高等教育课程体系以及学生培养的深度嵌合路径。”

**Example 2: 用户询问会议时间地点**
- **User Request**: “会议什么时候举行？地点在哪里？”
- **Agent Response**: “会议时间定于2026年7月3日（星期五）14:00 - 16:30。地点在上海财经大学国定路777号校区，具体会议室待定。”

**Example 3: 用户询问会议整体安排**
- **User Request**: “能告诉我会议的整体安排吗？”
- **Agent Response**: “好的，本次会议的整体安排如下：
    - 13:30 - 14:00 签到与入场
    - 14:00 - 14:10 开幕致辞
    - 14:10 - 14:50 主旨演讲（一）
    - 14:50 - 15:20 案例分享
    - 15:20 - 15:40 集体照&茶歇
    - 15:40 - 16:20 圆桌讨论
    - 16:20 - 16:30 总结与闭幕，自由交流”

## Common Issues
- **信息缺失处理不当**: 未能正确识别并告知用户文档中明确缺失的信息（如“具体会议室待定”），反而进行猜测或省略。
- **信息准确性偏差**: 提取会议主题、时间、地点时出现错别字或与原文不符的情况。
- **超出能力范围**: 错误地尝试回答关于会议报名、嘉宾联系方式或会议资料等超出本技能边界的问题。
- **混淆技能**: 将查询特定环节详情的请求误认为整体概览请求，未能正确路由到 `query-session-details`。
