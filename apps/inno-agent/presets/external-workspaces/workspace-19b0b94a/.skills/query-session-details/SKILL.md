# 查询会议环节详情
## When to use
*   用户询问特定时间段的活动内容，例如：“14:00到14:10是什么环节？”
*   用户查询某个演讲或分享的具体主题，例如：“主旨演讲的题目是什么？”
*   用户想知道某个环节的主讲嘉宾或分享嘉宾，例如：“谁是案例分享的主讲嘉宾？”
*   用户询问圆桌讨论的具体主题或参与嘉宾。
*   用户查询特定人物的演讲内容或分享主题。

## When not to use
*   用户询问会议的整体主题、时间或地点，应路由至 `get-conference-overview`。
*   用户请求会议报名、联系嘉宾、预定会议室等实时互动或操作，应拒绝并说明边界。
*   用户查询议程中未明确提及或标明“待定”的信息，例如：“会议室具体是哪个？”
*   用户询问会议资料、录像或嘉宾个人隐私信息，应拒绝。
*   用户请求超出会议议程范围的任何信息。

## Instructions
*   分析用户请求，识别其查询的是具体时间、活动、主题、嘉宾或环节类型。
*   从提供的会议议程文档中精确提取匹配的信息。
*   如果用户请求的信息在文档中明确缺失或标记为“待定”，则如实告知信息缺失。
*   以清晰、简洁的中文文本形式组织回复。

## Workflow
1.  **理解用户意图**: 识别用户是查询特定时间段的活动、主讲嘉宾、演讲主题或圆桌讨论内容。
2.  **信息检索**: 根据识别出的意图，在会议议程文档中查找对应的活动名称、时间、主题和嘉宾信息。
3.  **结果呈现**: 将检索到的信息组织成易于理解的中文回复。
4.  **处理边界情况**: 如果信息缺失或超出能力范围，则根据 `boundaries` 规则进行处理。

## Output Format
*   场景判断: 用户查询特定会议环节的详细信息。
*   处理方案: 从会议议程中提取对应时间、活动名称、主题、主讲嘉宾或分享嘉宾列表。
*   可用话术: "在[时间段]，活动是[活动名称]，主题是[主题]，主讲嘉宾是[嘉宾姓名]。" 或 "圆桌讨论的主题是[主题]，分享嘉宾有[嘉宾列表]。"
*   风险提示: 如果信息缺失或超出能力范围，需明确告知用户。
*   后续评估: 用户是否获得了准确且完整的环节详情信息。

## Examples
*   **用户输入**: "14:10到14:50是什么环节？"
*   **模型输出**: "在14:10到14:50，将进行主旨演讲（一），主题是《'应'变'而生'，以'融'促新：生成式AI时代数字经济人才培养探索》，主讲嘉宾是王子晔教授。"

*   **用户输入**: "谁是案例分享的主讲嘉宾？"
*   **模型输出**: "案例分享的主讲嘉宾是西班牙IE大学的万家阳。"

*   **用户输入**: "圆桌讨论的主题是什么？"
*   **模型输出**: "圆桌讨论的主题是“中外高校数智教育合作的机遇、挑战与机制创新”。"

## Common Issues
*   用户请求的信息在议程中不存在或未明确说明。
*   用户请求的信息被标记为“待定”，例如：“具体会议室待定”。
*   用户请求的信息属于其他技能的职责范围，例如询问会议总时长。
