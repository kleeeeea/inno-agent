# 需求转译与代码生成

## When to use
- 教师首次提出需要某类课堂工具（点名、倒计时、随机分组、学科演示等）
- 教师提供学生名单、学科知识点或视觉风格偏好，要求生成可运行文件
- 需要将教学创意（如"跳出名字的转盘"）转化为结构化技术实现（HTML+CSS+JS）

## When not to use
- 教师要求修改已生成工具的字体、配色、动画效果或按钮位置 → 转交 `interactive-optimizer`
- 教师询问如何保存文件、如何在教室电脑打开或如何全屏显示 → 转交 `interactive-optimizer`
- 教师仅进行纯教学咨询，无需生成任何工具 → 直接回答，不调用技能
- 对话中出现学生自伤、校园暴力、教师严重情绪危机等信号 → 立即停止生成，提供危机干预转介建议

## Instructions
- **需求澄清**：通过对话确认功能类型、班级规模、视觉风格（科技感/卡通风/护眼色）、交互细节（音效/动画/触摸要求）
- **约束检查**：确保生成为单文件HTML（内嵌CSS/JS），无后端依赖，仅使用CDN公开库（如Tailwind/Vue3），适配大屏触摸（按钮≥48px）
- **风险拦截**：拒绝预置学生照片/身份证号等隐私数据（建议仅保留姓名或现场拖拽图片），拒绝多人在线同步/数据库等需服务器的功能（降级为单机版）
- **学科保真**：调用学科逻辑模板确保数学/地理等计算准确（如经纬度转换、二进制算法）

## Workflow
1. **需求对齐**：使用鼓励性话术确认四个维度——功能类型、参与对象（人数/名单）、视觉风格、交互细节
2. **逻辑映射**：评估复杂度——简单工具直接生成，学科专用工具调用特定模板，复杂需求必须简化为单文件可承载范围
3. **实时生成**：输出V1.0单文件HTML代码块，含中文注释，关键逻辑说明，预置示例数据或名单接口
4. **安全提示**：明确告知隐私保护建议与离线运行优势，说明技术边界（无法多人实时同步）

## Output Format
- **场景判断**：需求复杂度等级（简单工具/学科专用/需降级处理）及技术可行性评估
- **处理方案**：选用的生成策略（基础组件/学科逻辑模板）与功能边界说明
- **生成代码**：完整单文件HTML代码块（内嵌CSS/JS，中文注释，离线可运行）
- **风险提示**：隐私数据保护建议（避免硬编码敏感信息）、网络依赖警告、教室环境适配说明
- **后续评估**：询问教师功能逻辑是否符合预期，引导至 `interactive-optimizer` 进行视觉微调与部署指导

## Examples
- **正面**："我想给三年级二班做个点名网页，名单如下..." → 生成带随机算法的点名器，名单预置在JS数组中，按钮适配触摸
- **正面**："需要地理课用的经纬度演示器，能切换南北半球" → 调用地理逻辑模板，确保经纬度计算准确，界面适配大屏
- **风险处理**："做个能实时同步30人答题数据的竞赛系统" → 拒绝服务器依赖需求，建议降级为单机版轮流操作或分屏展示

## Common Issues
- **隐私泄露风险**：教师要求预置学生照片、身份证号 → 仅保留姓名，建议图片改为现场拖拽加载，解释HTML文件易被复制的风险
- **网络依赖风险**：要求接入在线数据库、API验证 → 改为本地存储或手动输入模式，提示教室网络不稳定可能导致教学事故
- **技术过度复杂**：要求多人在线同步、用户登录系统、后端管理 → 坚决降级为单文件单机版，解释"即开即用"的技术边界
- **学科逻辑错误**：如二进制转换算法、地理坐标计算错误 → 内置标准学科逻辑模板，生成前验证算法准确性
