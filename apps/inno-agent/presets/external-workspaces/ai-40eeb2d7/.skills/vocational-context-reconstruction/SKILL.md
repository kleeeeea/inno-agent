# 职高化重构建议

## When to use
- 用户明确要求改造现有教案："这个教案不适合我的职高学生，怎么改？"
- 用户询问具体替换方案："应该怎么替换这个活动"、"有没有更好的导入方式"
- 用户需要职高专属适配方案："给一份适合120分钟连排课的实操方案"、"针对普高转来的学生设计分层任务"
- 诊断已完成，用户进入改造阶段，需要可立即落地的职高化实施细节

## When not to use
- **纯诊断需求**：若用户仅要求"先帮我诊断一下问题"、"看看这份教案哪里不对"，应转交 `lesson-plan-logic-diagnosis` 技能执行四阶校验
- **安全红线场景**：若发现考纲重大偏离（如大学算法/遗漏OS模块/异常处理）或涉及普高转学生心理安全陷阱（如"证明你比普高差"的对比活动），优先执行安全转介（提示"建议人工复核课标"/"建议班主任协同"），暂停重构
- **全新生成**：若用户要求从零开始编写教案而非改造现有内容，使用普通问答模式

## Instructions
1. **重拿轻放原则**：用真实编程任务替代伪信息化活动（如以"AI代码毒瘤诊所"替代"投纸片选工资"），确保每个环节直接服务于Python考纲知识点
2. **分层双轨设计**：针对普高转职高生（理论强化路径）与原生职高生（实操入门路径）设计A/B任务卡，避免"一刀切"导致挫败或无聊
3. **120分钟硬节奏**：严格遵循"20分钟模仿→30分钟改写→40分钟创造→30分钟互评"的实训节拍，确保纯理论讲解不超过20分钟连续时长
4. **防AI偷懒机制**：植入结构化角色分工（AI检索员、记录员、汇报员），要求提交过程性证据（搜索关键词截图、人工排查步骤），防止"碰到AI就不学了"

## Workflow
1. **学情锚定**：确认目标班级构成（普高转学生比例、原生职高生基础），标记需重构的高风险环节（幼稚化活动/纯理论段落）
2. **活动替代**：将伪信息化/低幼化设计替换为真实情境任务（如Bug修复、需求分析、代码审查），明确对接Python考纲具体条目
3. **分层卡设计**：构建A卡（基础模仿，保底完成）与B卡（进阶改写，跳级挑战），设置清晰的难度切换信号（如"发现AI代码3处以上错误可领B卡"）
4. **流程重组**：按120分钟连排课重新切割时间节点，确保"做中学"时间占比≥70%，插入防切屏检查点（每30分钟提交截图）
5. **管理结构植入**：添加小组角色分工表与过程性评价量规，明确"人可以问AI，但必须解释AI给出的方案"

## Examples

### Example 1: 替换伪信息化导入活动
**用户输入**："原来教案用'投纸片选工资区间'来讲if-elif，怎么改？"

**输出**：
```markdown
### 重构对象
投纸片选工资区间（导入环节）[标签：伪信息化/学情不适配]

### 替代方案
**活动名称**：AI代码毒瘤诊所
**对接考纲**：Python程序结构-条件语句（if-elif-else）
**设计意图**：用真实错误代码替代纸质游戏，培养"质疑AI"的职业素养
**操作步骤**：
1. 教师展示AI生成的"工资计算器"代码（含3处逻辑/语法错误）
2. 小组分工：AI检索员（查官方文档验证语法）、记录员（截图错误点）、汇报员（讲解修复思路）
3. 在真实Python环境中运行并修复，提交修复前后对比截图
**所需资源**：Python IDE、预置错误代码卡、在线语法文档
**过程性产出**：错误点标记截图、修复后的可运行代码、汇报员讲解视频（1分钟内）

### 分层策略
- **A卡（基础层）**：修复2处明显语法错误（如缩进、冒号遗漏），运行通过即可
- **B卡（进阶层）**：优化AI代码逻辑（将多重if改为if-elif结构），并添加输入异常处理（try-except）
- **切换机制**：A卡完成后，向教师展示"这样写有什么问题"的判断过程，即可领取B卡

### 120分钟流程优化
- **0-20min 模仿阶段**： 教师演示"发现AI代码错误→查文档→修复"的完整流程，学生跟做1个简单案例
- **20-50min 改写阶段**： 小组领取3个不同场景的AI错误代码卡（工资计算/成绩评定/快递计费），完成A卡修复
- **50-90min 创造阶段**： 小组互相编写"带毒瘤代码"并交换修复，尝试完成B卡优化（如添加异常处理）
- **90-120min 互评阶段**： 展示最难发现的逻辑错误及修复方案，评选"最佳诊所小组"，教师点评常见考纲失分点

### 管理风险提示
- **潜在问题**：学生直接让AI修复而非自己思考；普高转学生觉得A卡太简单感到被轻视
- **防范措施**：要求记录员必须提交"人工排查步骤"截图（如圈出错误行的手写批注）；对普高转学生私下提供B卡入口，公开话术统一为"有预习基础的同学可以直接挑战进阶任务"
```

## Common Issues
1. **活动过度游戏化**：为追求"趣味性"设计复杂规则（如积分闯关），反而分散编程注意力，应坚持"真实调试任务即最好游戏"
2. **分层标签化**：A卡B卡仅换数据不换认知深度，导致分层失效，需确保B卡涉及"改写逻辑结构"而非"换数字计算"
3. **时间估计乐观**：低估代码调试所需时间，导致120分钟流程前松后紧，需在"创造阶段"预留40分钟冗余应对卡壳
4. **角色分工形式化**：小组合作变成一人包办，需设计"角色专属产出物"（如检索员必须提交搜索关键词列表，记录员必须提交错误对比表）确保全员参与
