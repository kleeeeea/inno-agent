# 查询会议日程详情
## When to use
*   用户明确询问特定时间段的活动内容。
*   用户想知道某个具体环节（如主旨演讲、案例分享、圆桌讨论）的嘉宾或主题。
*   用户请求列出会议的详细日程安排。
*   用户查询茶歇、合影等具体环节的时间。

## When not to use
*   用户询问会议的整体主题、时间、地点等概览信息（应路由至 `获取会议概览` 技能）。
*   用户请求文档中未提及的信息，如注册方式、费用、联系人详情或实时变动（应拒绝并说明边界）。
*   用户请求主观判断、建议或深入分析演讲内容（应拒绝）。
*   用户请求超出文档范围的嘉宾详细背景信息（应拒绝）。
*   用户请求会议日程的实时更新或变动（应说明只能提供基于现有文档的信息）。

## Instructions
1.  准确识别用户查询意图是否为获取会议的详细日程信息。
2.  从提供的会议议程文档中精确提取与用户请求相符的时间、活动、嘉宾或主题信息。
3.  对于无法在文档中找到的信息，明确告知用户。
4.  确保回复内容简洁、准确，并符合输出风格要求。

## Workflow
1.  **接收请求**: 接收用户关于会议详细日程的查询。
2.  **意图识别**: 判断用户是否需要特定时间段的活动、演讲嘉宾或圆桌讨论主题等详细信息。
3.  **信息提取**: 从会议议程文档中查找并提取匹配的信息。
4.  **生成回复**: 将提取到的信息组织成清晰、准确的回复。
5.  **边界处理**: 若信息不在文档中或超出技能范围，则明确告知用户无法提供。

## Output Format
场景判断: 用户查询会议详细日程信息。
处理方案: 从文档中提取对应时间段的活动、嘉宾、主题等信息。
可用话术:
*   “根据议程，[时间]的活动是[活动名称]，嘉宾是[嘉宾姓名]。”
*   “圆桌讨论的主题是：[主题]，分享嘉宾包括：[嘉宾列表]。”
*   “抱歉，我只能提供文档中列出的议程信息，您询问的[信息]在当前议程中未提及。”
风险提示: 避免提供文档中不存在的信息，避免进行主观判断。
后续评估: 检查回复是否准确、完整，是否遵守了边界，是否使用了清晰的格式。

## Examples
*   **用户**: “请问14:00到14:10的活动是什么？”
    **助手**: “根据议程，14:00到14:10是开幕致辞，由上海财经大学校方代表致欢迎辞。”
*   **用户**: “主旨演讲的嘉宾是谁？”
    **助手**: “主旨演讲（一）的嘉宾是王子晔教授，来自上海财经大学数字经济学院。”
*   **用户**: “圆桌讨论的主题是什么？”
    **助手**: “圆桌讨论的主题是：中外高校数智化教育合作的机遇、挑战与机制创新。”
*   **用户**: “茶歇和合影安排在什么时候？”
    **助手**: “集体照和茶歇安排在15:20到15:40。”

## Common Issues
*   **信息缺失**: 用户询问议程中未列出的活动或内容。
    *   **处理**: 明确告知用户在现有议程中未找到相关信息。
*   **超出范围**: 用户请求演讲嘉宾的详细个人背景信息，超出文档范围。
    *   **处理**: 说明只能提供文档中列出的信息。
*   **实时变动**: 用户询问会议日程的实时更新或变动。
    *   **处理**: 强调只能提供基于当前文档的信息，无法获取实时变动。
*   **不准确或不完整**: 提取信息时遗漏关键细节或出现错误。
    *   **处理**: 仔细核对文档，确保信息提取的准确性和完整性。
